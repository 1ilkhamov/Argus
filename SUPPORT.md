# Support

## Где искать ответы сначала

Перед тем как открывать issue, проверьте:

- `README.md`
- `docs/README.md`
- `docs/backend.md`
- `docs/frontend.md`
- `docs/operations.md`
- `docs/soul-customization.md`

## Когда открывать issue

Используйте GitHub Issues, если у вас:

- воспроизводимый баг
- feature request
- проблема в документации
- сбой в CI, сборке или onboarding flow

Если проблема похожа на баг, приложите:

- шаги воспроизведения
- ожидаемое поведение
- фактическое поведение
- логи или stack trace
- информацию об окружении

## Что не публиковать публично

Не публикуйте в issue:

- токены
- API keys
- `.env` содержимое
- приватные данные пользователей
- эксплуатационные детали security-проблем

## Security issues

Для security-вопросов не используйте публичные issues.

См. `SECURITY.md`.

## Contribution flow

Если вы хотите прислать исправление или улучшение:

- сначала посмотрите `CONTRIBUTING.md`
- затем откройте issue или сразу подготовьте PR, если изменение небольшое и очевидное
